# Peak Workout App

Python serves the frontend and API. Supabase provides Postgres and authentication, Resend sends authentication email, and Open Food Facts provides product details.

## Local development

1. Copy `.env.example` to `.env` and set the three Supabase values.
2. Install dependencies with `pip install -r requirements.txt`.
3. Run the SQL exposed at `/api/schema` in the Supabase SQL Editor.
4. Start the server with `python server.py`.
5. Open `http://localhost:8000/test`.

### Full Supabase schema refresh

The complete additive schema is stored in
`supabase/migrations/202607060001_full_application_schema.sql`. Start the server,
open `http://localhost:8000/api/supabase/schema`, and paste that SQL into the
Supabase SQL Editor. It creates only missing application tables and supporting
objects and never drops a table.

After applying the SQL, run:

```powershell
python server.py --refresh-supabase-full
```

The command verifies that the SQL was applied, synchronizes `body_metrics` and
`exercise_metrics`, and refreshes the exercise catalog. `--refresh` is a shorter
alias. `public.foods` remains the lightweight code/name search index, and logged
meal items reference its `code`. Full product and nutrition details are not copied
into Supabase; they are resolved from Open Food Facts by that code.

Refreshing the code/name index remains a separate operation because it requires
the local source file:

```powershell
python server.py --refresh-foods "C:\path\food.parquet"
```

### Body metric catalog

`metrics.xlsx` is the source catalog for `public.body_metrics`. It contains 321 unique metric definitions with `id`, `name`, `dimension`, and `category`.

1. Run the SQL from `http://localhost:8000/api/metrics/schema` in the Supabase SQL Editor once.
2. Import the workbook with `python server.py --refresh-metrics`.
3. Re-running the command safely upserts the same IDs instead of creating duplicates.

An explicit HTTP sync is also available as `POST /api/metrics/sync` for local testing. Protect or disable it before production, as noted below.

### Exercise metric capability matrix

`public.exercise_metrics` maps each equipment type to twelve boolean tracking capabilities. Apply the SQL from `/api/exercise-metrics/schema`, then synchronize the 14 approved rows with:

```powershell
python server.py --refresh-exercise-metrics
```

Use `localhost`, not `127.0.0.1`, when testing passkeys. Cookies and passkeys treat those as different hosts.

## Production domain source of truth

Choose the final application hostname before enrolling real passkeys. Replace the placeholders below once that decision is made.

| Purpose | Local value | Suggested production value |
|---|---|---|
| Application origin | `http://localhost:8000` | `https://<APP_HOST>` |
| Application host | `localhost` | e.g. `app.riddhimandal.com` |
| Application API | `http://127.0.0.1:8000/api` | `https://<APP_HOST>/api` |
| Passkey RP ID | `localhost` | `<APP_HOST>` or `riddhimandal.com`; see below |
| Passkey origin | `http://localhost:8000` | `https://<APP_HOST>` |
| Authentication sender | `no-reply@mail.riddhimandal.com` | Already production-ready |
| Supabase API URL | `https://<project-ref>.supabase.co` | Keep this, unless enabling a Supabase custom domain |

Do not use `auth.riddhimandal.com` as the passkey RP ID unless the browser UI that creates and uses passkeys is actually hosted on that hostname.

### Decide the purpose of `auth.riddhimandal.com`

The hostname can remain unused until production architecture is finalized. Assign it only one CNAME destination:

- If the Python login UI is deployed at `auth.riddhimandal.com`, treat that as `<APP_HOST>` throughout this checklist and use it as the passkey RP ID.
- If it becomes a paid Supabase custom domain, point it to Supabase, update `SUPABASE_URL` and the frontend CSP, and host the application UI somewhere else.
- Do not point the same hostname to both Heroku and Supabase. A DNS CNAME can have only one destination.
- It is unrelated to the Resend sending domain `mail.riddhimandal.com`.

## Production domain migration checklist

### 1. Heroku and Hostinger DNS

- [ ] Add the final hostname in Heroku under **Settings → Domains**.
- [ ] Copy the DNS target generated by Heroku.
- [ ] In Hostinger's `riddhimandal.com` DNS zone, create the requested CNAME for `<APP_HOST>` pointing to Heroku's generated target.
- [ ] Wait for Heroku Automated Certificate Management to issue HTTPS before enabling production logins.
- [ ] Keep the existing Resend records under `send.mail` and `resend._domainkey.mail`; the application CNAME must not replace them.
- [ ] Keep Titan Mail's root `@` MX records if `@riddhimandal.com` mailboxes are still used.

The checked-in `Procfile` already binds the Python server to Heroku's `$PORT` on `0.0.0.0`.

### 2. Supabase Authentication URL configuration

In **Supabase → Authentication → URL Configuration**:

- [ ] Change **Site URL** from `http://localhost:8000/auth-test` to `https://<APP_HOST>/` or the final login route.
- [ ] Add `https://<APP_HOST>/**` to **Redirect URLs**.
- [ ] Remove local redirect URLs when production no longer needs them, or retain them only for intentional local development.
- [ ] Review confirmation, recovery, invitation, magic-link, and email-change templates. Prefer Supabase template variables such as `{{ .ConfirmationURL }}` instead of hard-coded localhost links.
- [ ] If CAPTCHA is enabled later, add `<APP_HOST>` to that provider's allowed-domain list.

Existing confirmation emails keep the redirect embedded when they were sent. Send a new email after changing URL settings.

### 3. Supabase passkeys / WebAuthn

In **Supabase → Authentication → Passkeys**:

- [ ] Set **Relying Party Display Name** to the final product name.
- [ ] Replace RP ID `localhost` with the stable production RP ID.
- [ ] Replace origin `http://localhost:8000` with the exact HTTPS origin `https://<APP_HOST>`; do not include a path or trailing slash.
- [ ] Test registration and sign-in from the production hostname.

RP ID selection is permanent from the user's perspective. Changing it makes every previously enrolled passkey unusable:

- Use `<APP_HOST>` when passkeys should work only on that hostname.
- Use `riddhimandal.com` only when passkeys intentionally need to work across multiple `*.riddhimandal.com` origins, and list every allowed origin explicitly in Supabase.
- Localhost passkeys are development-only and will not work in production.

Reference: [Supabase passkey configuration](https://supabase.com/docs/guides/auth/passkeys).

### 4. Supabase SMTP and Resend

The verified sending domain `mail.riddhimandal.com` is already suitable for production and does not depend on the application hostname.

- [ ] In Resend, keep `mail.riddhimandal.com` verified.
- [ ] In Supabase under **Authentication → Notifications → Email → SMTP Settings**, confirm:

  ```text
  Sender email: no-reply@mail.riddhimandal.com
  Sender name: Peak
  Host: smtp.resend.com
  Port: 465
  Username: resend
  Password: a domain-restricted Resend sending key
  ```

- [ ] Confirm SPF, DKIM, MX/return-path, and DMARC records remain valid in Hostinger.
- [ ] If Resend webhooks are added for delivery or bounce events, replace their localhost URL with `https://<APP_HOST>/<webhook-path>`.
- [ ] Send confirmation, recovery, and invitation test messages and check their links and Resend delivery logs.
- [ ] Never put the Resend API key in frontend code or commit it to Git.

Reference: [Resend with Supabase](https://resend.com/docs/knowledge-base/getting-started-with-resend-and-supabase).

### 5. Supabase API/custom domain

No change is required if `SUPABASE_URL` continues to use the project's `*.supabase.co` address.

If a paid Supabase custom domain such as `api.riddhimandal.com` is enabled:

- [ ] Change `SUPABASE_URL` in Heroku config and local `.env`.
- [ ] Update `connect-src` in `frontend/auth-test.html`; it currently permits `'self'` and `https://*.supabase.co`, not an arbitrary custom Supabase hostname.
- [ ] Update OAuth provider callback URLs, SAML metadata, mobile deep links, and any external integrations that reference the old Supabase project URL.
- [ ] Re-test password login, token refresh, passkeys, and database requests.

Reference: [Supabase custom domains](https://supabase.com/docs/guides/platform/custom-domains).

### 6. Repository and Heroku configuration

Set these as Heroku config vars; never commit their values:

```text
SUPABASE_URL=https://<project-ref>.supabase.co
SUPABASE_SECRET_KEY=<server-only secret key>
SUPABASE_PUBLISHABLE_KEY=<browser-safe publishable key>
PEAK_COOKIE_SECURE=1
```

Additional settings:

- [ ] Set `GYM_TRACKER_API_URL=https://<APP_HOST>/api` if `backend/sync_exercises.py` will call the deployed API. Its fallback is currently `http://127.0.0.1:8000/api`.
- [ ] Review `PEAK_REFRESH_COOKIE_SECONDS` for the desired production session duration.
- [ ] Rotate any key that was pasted into chat, logs, source code, or committed history.

The main authentication flow does not contain a hard-coded application hostname. It validates POST origins against the incoming `Host` header and enables secure cookies when Heroku supplies `X-Forwarded-Proto: https`. Setting `PEAK_COOKIE_SECURE=1` makes the production requirement explicit.

### 7. Production security blockers

Do not treat the current test server as production-ready until these are addressed:

- [ ] Disable or authenticate the generic `/api/<table>` mutation routes. They currently use the server's privileged Supabase key.
- [ ] Remove or protect schema endpoints such as `/api/schema`, `/api/users/schema`, and destructive migration endpoints.
- [ ] Remove or protect `/test`, `/auth-test`, refresh controls, and diagnostic pages that should not be public.
- [ ] Replace the in-process login attempt limiter with a shared limiter if running multiple workers/dynos.
- [ ] Keep Supabase RLS enabled and test policies using authenticated and anonymous clients.
- [ ] Configure Supabase leaked-password protection, CAPTCHA, recovery flows, and production rate limits as appropriate.
- [ ] Add monitoring for authentication errors, Resend bounces/complaints, and abnormal API traffic.
- [ ] Back up Supabase and test account recovery before launch.

## Final production verification

- [ ] `https://<APP_HOST>` has a valid certificate and redirects HTTP to HTTPS.
- [ ] Signup sends a Resend confirmation email from `mail.riddhimandal.com`.
- [ ] Confirmation returns to the production application, not localhost.
- [ ] Username and email login both work.
- [ ] Session refresh and logout work after a browser restart.
- [ ] A passkey enrolled on production can sign in on production.
- [ ] Localhost passkeys are not expected to work on production.
- [ ] No secret keys appear in browser source, JSON responses, Git history, or logs.
- [ ] Unauthorized users cannot read or mutate another user's data.
